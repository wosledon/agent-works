# .NET 博客系统 - 测试计划 (Test Plan)

**项目**: one-blog  
**版本**: v1.0.0  
**日期**: 2026-03-14  
**QA 负责人**: 测试工程师  

---

## 1. 项目概述

### 1.1 项目背景
one-blog 是一个基于 .NET 技术栈构建的博客系统，包含文章管理、用户认证、评论系统等核心功能。

### 1.2 技术架构
- **后端**: .NET 8 Web API
- **前端**: React + TypeScript
- **数据库**: PostgreSQL
- **缓存**: Redis

---

## 2. 测试目标

### 2.1 功能目标
- 确保核心业务流程正确运行
- 验证用户认证和权限控制
- 确认数据持久化和一致性

### 2.2 非功能目标
- 接口响应时间 < 500ms (P95)
- 并发用户支持 100+
- 系统可用性 ≥ 99.9%

## 4. 质量门禁 (Quality Gates)

### 4.1 代码质量门禁
```
┌─────────────────────────────────────────────────────┐
│  阶段          │  检查项           │  门禁标准       │
├─────────────────────────────────────────────────────┤
│  代码提交      │  单元测试覆盖      │  ≥ 70%         │
│                │  代码扫描          │  0 高危漏洞     │
├─────────────────────────────────────────────────────┤
│  集成阶段      │  集成测试通过      │  100%          │
│                │  API 契约测试      │  全部通过       │
├─────────────────────────────────────────────────────┤
│  预发布        │  E2E 测试通过      │  关键路径 100%  │
│                │  性能测试          │  P95 < 500ms   │
├─────────────────────────────────────────────────────┤
│  发布前        │  缺陷检查          │  无 P0/P1 缺陷  │
│                │  回归测试          │  100% 通过      │
└─────────────────────────────────────────────────────┘
```

### 4.2 缺陷严重级别定义
- **P0 - 致命**: 系统崩溃、数据丢失、安全漏洞
- **P1 - 严重**: 核心功能不可用、数据错误
- **P2 - 一般**: 非核心功能异常、界面问题
- **P3 - 轻微**: 文案错误、样式偏差

## 5. 测试策略

### 5.1 测试金字塔

```
              /\
             /  \     E2E 测试 (10%)
            /    \    - 关键用户旅程
           /──────\   - Playwright
          /        \
         /  集成测试 \  (20%)
        /   - API    \  - WebApplicationFactory
       /   - 数据库   \ - TestContainers
      /────────────────\
     /                  \
    /    单元测试 (70%)   \
   /  - xUnit (.NET)      \
  /   - Vitest (React)     \
 /──────────────────────────\
```

### 5.2 测试类型

#### 5.2.1 单元测试
- **框架**: xUnit (.NET), Vitest (React)
- **目标**: 业务逻辑、工具函数、组件行为
- **工具**: Moq (模拟), FluentAssertions (断言)
- **覆盖率目标**: 70%+

#### 5.2.2 集成测试
- **框架**: WebApplicationFactory, TestContainers
- **目标**: 
  - API 端点与数据库交互
  - 服务层与外部依赖集成
  - 数据库迁移验证
- **工具**: PostgreSQL TestContainer

#### 5.2.3 端到端测试 (E2E)
- **框架**: Playwright
- **目标**: 
  - 关键用户旅程
  - 跨浏览器兼容性
- **浏览器**: Chromium, Firefox, WebKit

### 5.3 自动化测试流程

```
代码提交
   │
   ▼
┌─────────────┐
│ CI 流水线   │
│  ├─ 构建    │
│  ├─ 单元测试│ ← 70%+ 覆盖
│  └─ 代码扫描│
└─────────────┘
   │
   ▼
┌─────────────┐
│ 集成测试    │ ← WebApplicationFactory
│  ├─ API 测试│
│  └─ DB 测试 │
└─────────────┘
   │
   ▼
┌─────────────┐
│ E2E 测试    │ ← Playwright
│ (关键路径)   │
└─────────────┘
   │
   ▼
  发布
```

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 测试环境不稳定 | 延迟 | 使用 TestContainers 隔离 |
| 需求变更频繁 | 返工 | 与产品保持每日同步 |
| 自动化维护成本高 | 效率低 | 优先覆盖关键路径 |
| 性能测试不充分 | 上线风险 | 引入负载测试到 CI |
