# One-Report 测试计划

> 项目: one-report（低代码报表工具）
> 版本: v1.0.0
> 最后更新: 2026-03-14

---

## 1. 项目概述

one-report 是一款低代码报表工具，支持拖拽式报表设计、多数据源接入和自动化数据导出。本测试计划覆盖核心业务场景及关键技术指标验证。

---

## 2. 测试目标

### 2.1 功能目标
- 验证报表设计器的核心功能稳定性
- 确保数据查询与展示的准确性
- 验证多数据源配置与切换的正确性

### 2.2 性能目标
- **大文件导出性能**: 支持 100万+ 行数据导出，响应时间可接受
- **内存占用控制**: 导出过程中内存增长可控，无 OOM 风险
- **并发处理能力**: 多用户同时操作时系统稳定

### 2.3 兼容性目标
- 支持主流关系型数据库（MySQL、PostgreSQL、SQL Server、Oracle）
- 支持 NoSQL 数据源（MongoDB、Elasticsearch）
- 支持 API 数据源（RESTful、GraphQL）

## 4. 测试策略

### 4.1 大文件导出性能测试

**测试场景设计:**

| 场景 | 数据量 | 预期指标 |
|------|--------|----------|
| 基础导出 | 1万行 × 20列 | < 3秒 |
| 中等导出 | 10万行 × 20列 | < 15秒 |
| 大文件导出 | 100万行 × 20列 | < 120秒 |
| 超大文件导出 | 500万行 × 20列 | < 600秒，支持流式处理 |

**测试方法:**
1. 使用 JMeter/k6 模拟并发导出请求
2. 监控系统资源（CPU、内存、IO）
3. 记录导出时间、内存峰值、GC 频率
4. 验证导出文件完整性（MD5 校验）

**测试数据准备:**
```bash
# 生成测试数据脚本
curl -X POST http://test-env/seed-data \
  -H "Content-Type: application/json" \
  -d '{"rows": 1000000, "columns": 20, "types": ["string", "number", "date"]}'
```

### 4.2 内存占用监控

**监控指标:**

| 指标 | 工具 | 阈值 |
|------|------|------|
| Heap Memory | JVM Metrics / Node heapdump | < 80% 堆内存 |
| GC 暂停时间 | GC logs | < 200ms |
| 内存泄漏 | Leak detection | 0 泄漏 |
| 导出过程内存峰值 | Prometheus + Grafana | < 4GB |

**监控方案:**
1. **后端**: JVM Heap dump + JFR 录制
2. **前端**: Chrome DevTools Memory timeline
3. **系统**: Prometheus + Grafana 实时看板
4. **告警**: 内存使用 > 85% 触发告警

**内存测试用例:**
- TC-MEM-001: 连续导出 10 次大文件，观察内存回落情况
- TC-MEM-002: 并发 10 用户同时导出，验证内存分配
- TC-MEM-003: 长时运行测试（24小时），检查内存泄漏

### 4.3 多数据源兼容性

**支持的数据源矩阵:**

| 数据源 | 驱动版本 | 测试重点 | 优先级 |
|--------|----------|----------|--------|
| MySQL 8.0 | mysql-connector-java 8.0+ | 连接池、SSL、时区 | P0 |
| PostgreSQL 14+ | postgresql 42.x | JSONB 字段、数组类型 | P0 |
| SQL Server 2019 | mssql-jdbc 9.x | Windows 认证、分页 | P1 |
| Oracle 19c | ojdbc11 21.x | CLOB/BLOB、游标 | P1 |
| MongoDB 6.0 | mongodb-driver 4.x | Aggregation Pipeline | P1 |
| Elasticsearch 8.x | elasticsearch-java 8.x | 复杂查询、聚合 | P2 |
| REST API | - | 认证、超时、限流 | P1 |

**兼容性测试方法:**
1. **连接测试**: 验证各数据源的连接配置和驱动加载
2. **SQL 方言测试**: 验证分页、日期函数、类型转换
3. **数据类型映射**: 测试特殊类型（JSON、Array、Geometry）
4. **故障恢复**: 模拟网络中断、数据库重启后的重连

## 6. 测试用例

### 6.1 性能测试用例

#### TC-PERF-001: 大文件导出基准测试
```
前置条件: 数据库已准备 100万行测试数据
测试步骤:
  1. 登录系统，进入报表设计器
  2. 创建包含 20 个字段的报表
  3. 执行 Excel 导出
  4. 记录响应时间和内存占用
预期结果:
  - 导出时间 < 120秒
  - 内存峰值 < 4GB
  - 文件 MD5 校验通过
```

#### TC-PERF-002: 并发导出压力测试
```
前置条件: 准备 5 个不同数据量的报表
测试步骤:
  1. 使用 k6 模拟 20 并发用户
  2. 每个用户循环执行 10 次导出
  3. 监控服务器资源
预期结果:
  - 成功率 > 99%
  - 平均响应时间 < 60秒
  - 无 OOM 或连接池耗尽
```

### 6.2 内存测试用例

#### TC-MEM-001: 导出后内存回收验证
```
前置条件: JVM 启用 -XX:+HeapDumpOnOutOfMemoryError
测试步骤:
  1. 记录初始内存基线
  2. 连续执行 10 次大文件导出
  3. 触发 Full GC
  4. 记录最终内存
预期结果:
  - 内存回落到基线 ±10%
  - 无内存泄漏迹象
```

### 6.3 兼容性测试用例

#### TC-COMP-001: MySQL 数据源连接
```
前置条件: MySQL 8.0 实例运行
测试步骤:
  1. 配置 MySQL 连接参数
  2. 测试连接验证
  3. 执行查询和导出
  4. 验证时区和字符集
预期结果:
  - 连接成功
  - 查询结果正确
  - 中文无乱码
```

## 8. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 大文件导出 OOM | 高 | 实施流式处理、增加内存阈值告警 |
| 数据源驱动冲突 | 中 | 使用独立 ClassLoader 隔离 |
| 测试数据不足 | 中 | 提前准备数据生成脚本 |
| 性能环境不稳定 | 中 | 使用容器化环境确保一致性 |
