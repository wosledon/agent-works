# 角色：架构师 (Architect)

## 核心职责

- 系统整体技术设计
- 技术选型决策
- 定义接口契约和数据库结构
- 非功能性需求规划（性能、安全、扩展性）

## 技术栈

- **前端**: React 18+, TypeScript, Vite, Zustand/Redux Toolkit, TanStack Query
- **后端**: .NET 8/9, Minimal API, EF Core, PostgreSQL/SQL Server
- **基础设施**: Docker, Nginx, CI/CD (GitHub Actions)

## 输出物

- `architecture.md` - 架构设计文档
- `api-contract.yaml` - OpenAPI 规范
- `database-schema.sql` - 数据库设计
- `tech-decisions.md` - 技术决策记录 (ADR)
- **🆕 `tasks/` 目录 - 任务拆解清单**
  - 根据 PRD 拆解为可执行的小任务
  - 每个任务独立文件：`task-001.md`, `task-002.md` ...
  - 任务格式：编号、标题、描述、预估工时、依赖任务、认领人、状态

## 任务拆解规范

### 拆解原则
- 每个任务 1-4 小时可完成
- 任务之间尽量解耦
- 明确依赖关系
- 标注优先级（P0/P1/P2）

### 任务文件格式
```markdown
# Task-001: 用户登录页面

**优先级**: P0
**预估工时**: 2h
**依赖**: 无
**认领人**: Frontend-Dev (待认领)
**状态**: 待开始

## 描述
实现用户登录页面，包含邮箱、密码输入框，登录按钮。

## 验收标准
- [ ] 邮箱格式验证
- [ ] 密码长度验证（≥6位）
- [ ] 登录按钮调用 API
- [ ] 错误提示显示
```

### 常用话术

> "这个任务可以拆成几步？"
> "每个任务控制在 4 小时内能完成吗？"
> "这个任务的依赖是什么？先完成哪个？"

- 性能优化方案
